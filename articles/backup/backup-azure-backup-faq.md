# Azure 備份服務常見問題集
下列是關於 Azure 備份常見問題的清單。 如果您有關於 Azure 備份方案的任何其他問題，請移至 [論壇](https://social.msdn.microsoft.com/forums/azure/home?forum=windowsazureonlinebackup) 並張貼您的問題。 我們社群的服務人員將協助您找到答案。 若遇到常見問題，我們會將其加入此文章，以便您可以快速且輕鬆地找到。

## 安裝和設定
**Q1。 我可以備份至 Azure，使用 Azure 備份支援的作業系統清單為何？** <br/>
A1. Azure 備份支援下列作業系統清單


| 作業系統        | 平台           | SKU  |
| :------------- |-------------| :-----|
| Windows 8 和最新的 SP      | 64 位元 | Enterprise、Pro |
| Windows 7 和最新的 SP      | 64 位元 | Ultimate、Enterprise、Professional、Home Premium、Home Basic、Starter |
| Windows 8.1 和最新的 SP | 64 位元      |    Enterprise、Pro |
| Windows 10      | 64 位元 | 企業版、專業版、家用版 |
|Windows Server 2012 R2 和最新的 SP| 64 位元| Standard、Datacenter、Foundation|
|Windows Server 2012 和最新的 SP|    64 位元| Datacenter、Foundation、Standard|
|Windows Storage Server 2012 R2 和最新的 SP  |64 位元|    Standard、Workgroup|
|Windows Storage Server 2012 和最新的 SP |64 位元 |Standard、Workgroup
|Windows Server 2012 R2 和最新的 SP  |64 位元|    Essential|
|Windows Server 2008 R2 SP1 |64 位元|    Standard、Enterprise、Datacenter、Foundation|
|Windows Server 2008 R2    |64 位元|    Standard、Enterprise、Datacenter、Foundation|

**Q2。 哪裡可以下載最新版的 Azure 備份代理程式？** <br/>
A2. 您可以下載最新版的代理程式從 [這裡](http://aka.ms/azurebackup_agent)。 這可以安裝在 Windows Server、SCDPM 伺服器或 Windows 用戶端上

**Q3。 支援的 SCDPM 伺服器版本為何？** <br/>
A3。 我們建議您安裝 [最新](http://aka.ms/azurebackup_agent) SCDPM 的最新的更新彙總套件上的 Azure 備份代理程式 (2015 年 7 月後的為 UR6)

**Q4。 當設定 Azure 備份代理程式時，系統提示我要輸入「保存庫認證」。 那里任何到期日期附加至保存庫認證嗎？** <br/>
A4. 是，保存庫認證將於 48 小時後過期。 若檔案已過期，請登入 Azure 入口網站，並從您的備份保存庫下載保存庫認證檔。

**Q5。 是否有任何可以在每個 Azure 訂閱中建立的備份保存庫數目限制？** <br/>
A5. 是。 自 2015 年 7 月起，每個訂用帳戶您可以建立 25 個保存庫。 若您需要更多的保存庫，請建立新的訂用帳戶。

**Q6。 我應該將保存庫視為計費實體嗎？** <br/>
A6. 雖然可以取得每個保存庫的詳細計費，但強烈建議您考慮 Azure 訂用帳戶作為計費實體。 這在所有服務皆保持一致，而且您可以更輕鬆地進行管理。

**Q7。 是否有任何限制可以針對每個保存庫註冊的伺服器/電腦數目？** <br/>
A7。 是，您可以針對每個保存庫註冊最多 50 個電腦。 對於 Azure IaaS 虛擬機器，限制為每個保存庫 100 個 VM。 若您需要註冊更多電腦，請建立新的保存庫。

**Q8。 是否有任何限制可以從 Windows server/用戶端或 SCDPM 伺服器備份的資料量？** <br/>
A8。 編號

**Q9。 如何註冊我的伺服器到另一個資料中心？**<br/>
A9. 一般情況下，備份資料會傳送至「備份服務」的資料中心以進行註冊。 若要變更資料中心，最簡單的方式是解除安裝代理程式，並重新安裝代理程式，然後註冊新的資料中心。

**Q10。 重新命名的 Windows 伺服器備份至 Azure 的資料有何影響？** <br/>
A10. 任何目前已設定的備份將會停止。 您需要使用備份保存庫重新註冊伺服器，且復原服務會將其視為新的伺服器，因此註冊後進行的第一次備份作業，將會是包含在備份中所有資料的完整備份，而非僅包含自上次進行備份後的變更。 不過，如果您需要執行復原作業，您可以使用復原服務，從其他伺服器復原選項來復原已備份的資料。

**Q11。 哪種磁碟機我可以備份檔案和資料夾？** <br/>
A11. 下列的磁碟機/磁碟區組合無法進行備份：

- 卸除式媒體：必須報告為固定式磁碟機，才能作為備份項目來源使用。
- 唯讀磁碟區：必須為可寫入磁碟區，才能使磁碟區陰影複製服務 (VSS) 正常運作。
- 離線磁碟區：必須為線上磁碟區，才能使 VSS 正常運作。
- 網路共用：必須為本機磁碟區，才能使用線上備份來備份伺服器。
- 受 Bitlocker 保護的磁碟區：磁碟區必須先解除鎖定才能執行備份。
- 檔案系統識別碼：NTFS 是唯一支援這個版本的線上備份服務的檔案系統。

**Q12。 哪些檔案和資料夾的類型可以備份從我的伺服器？**<br/>
A12. 支援下列類型：

- 已加密
- 已壓縮
- 疏鬆
- 已壓縮 + 疏鬆
- 永久連結：不支援並略過
- 重新分析點：不支援並略過
- 已加密 + 已壓縮：不支援並略過
- 已加密 + 疏鬆：不支援並略過
- 已壓縮資料流：不支援並略過
- 疏鬆資料流：不支援並略過

**Q13。 什麼是快取資料夾的最低大小需求？** <br/>
A13。 快取資料夾的大小取決於您正在備份的資料量。 一般而言，您應該預期系統會將資料儲存空間所需的 10-15% 空間配置給快取資料夾。

**Q14。 如何隔離特定伺服器資料，從我的組織中的其他伺服器復原？**<br/>
A14. 使用相同保存庫註冊的任何伺服器，可以復原由使用相同複雜密碼之其他伺服器所備份的資料。 針對伺服器，若您想要確保復原只會在組織中的特定伺服器進行，您應該使用不同的複雜密碼來指派給這些伺服器。 例如，人力資源伺服器可能使用一組加密複雜密碼，而會計伺服器使用另一組，並且儲存體伺服器使用第三組。

**Q15。 我可以 「 移轉 」 備份資料之間的訂閱嗎？** <br/>
A15 ︰ 否

**Q16。 我可以 「 移轉 」 備份保存庫訂用帳戶之間嗎？** <br/>
A16 ︰ 否。 保存庫會建立在訂閱層級，而且無法重新指派至另一個訂用帳戶建立之後。

**Q17。 Azure 備份代理程式上運作的伺服器，使用 Windows Server 2012 重複資料刪除嗎？** <br/>
A17: [是]。 當代理程式服務準備備份作業時，會將重複資料刪除的資料轉換成一般資料。 它接著會最佳化資料以備份，加密資料，然後將加密的資料傳送至線上備份服務。

**Q18。 沒有如果我在啟動後取消備份會刪除備份資料？** <br/>
A18 ︰ 否。 備份保存庫取消點已傳輸的資料儲存的備份。 Azure 備份使用檢查點機制，讓備份資料在備份期間偶爾會取得檢查點，並在下一個備份程序驗證檔案的完整性。 下次觸發的備份會隨先前經備份的資料遞增。 這提供較佳的頻寬使用率，讓您不需要重複傳輸相同的資料。

**Q19。 為什麼即使我之前已經排定定期備份看到 「 Azure 備份尚未設定此伺服器 」 警告？** <br/>
A19 ︰ 這會儲存在本機伺服器上的備份排程設定不是儲存在備份保存庫中的設定值相同。 當伺服器或設定已復原至已知的良好狀態時，備份排程可能會失去同步處理。 如果發生這種情況，您應該重新設定備份原則，然後 **執行立即備份** 重新同步處理本機伺服器與 Azure。

**Q20。 防火牆規則都設定為備份的 Azure 備份？** <br/>
A20。 請確定防火牆規則啟用與下列 URL 的通訊，以進行 Azure 內部部署的無縫式備份與 Azure 上的工作負載保護：

- www.msftncsi.com
- \*.Microsoft.com
- \*.WindowsAzure.com
- \*.microsoftonline.com
- \*.windows.net


## 備份和保留
**Q1。 每個要備份的資料來源的大小是否有限制？** <br/>
A1. 下面將說明截至 2015 年 8 月，適用於各種作業系統的資料來源大小上限

|S.No | 作業系統 |  資料來源的大小上限 |
| :-------------: |:-------------| :-----|
|1| Windows Server 2012 或更新版本| 54400 GB|
|2| Windows 8 或更新版本| 54400 GB|
|3| Windows Server 2008、Windows Server 2008 R2 | 1700 GB|
|4| Windows 7 | 1700 GB|

資料來源大小是以下列所述的方式來測量

|   資料來源  |   詳細資料 |
| :-------------: |:-------------|
|磁碟區 |從電腦單一磁碟區備份的資料量。 這適用於在伺服器和用戶端電腦上受到保護的磁碟區。|
|Hyper-V 虛擬機器|要備份的虛擬機器上所有 VHD 的資料總和|
|Microsoft SQL Server 資料庫|要備份的單一 SQL 資料庫大小的大小 |
|Microsoft SharePoint|在要備份的 SharePoint 伺服陣列中內容和設定資料庫的總和|
|Microsoft Exchange|在要備份的 Exchange Server 中所有 Exchange 資料庫的總和|
|BMR/系統狀態|在要備份的電腦上 BMR 或系統狀態的每個個別複本|

**Q2。 是每日可以排程的備份次數有限制？**<br/>
A2. 是，Azure 備份透過 Windows Server/用戶端每日可以備份 3 個複本，透過 SCDPM 每日可以備份 2 個複本，而透過 IaaS VM 每日可以備份 1 個。

**Q3。 有可與 DPM 的 Azure 備份 （也就是差異 在沒有 DPM 的 Windows 伺服器） 上的備份排程原則？** <br/>
A3。 是。 您可以使用 DPM 指定每日、每週、每月、每年排程，或從 Windows Server (不包含 DPM) 指定僅每日和每週排程。

**Q4。 有可與 DPM 的 Azure 備份 （也就是差異 在沒有 DPM 的 Windows 伺服器） 上保留原則？**<br/>
A4. 否，您擁有相同的功能。 您可以指定每日、每週、每月和每年保留原則。

**Q5。 可以設定我保留原則選擇 – 也就是設定每週和每日，但不是每年和每月嗎？**<br/>
A5. 是的，Azure 備份保留結構讓您可以依照您的需求，彈性定義保留原則。

**Q6。 我可以在下午 6: 「 排程備份 」 並在不同的時間指定 「 保留原則 」 嗎？**<br/>
A6. 編號 保留原則只能套用在復原點上。 在下列影像中，保留原則會被備份指定 12 am 和下午 6。 <br/>

**Q7。 增量複本被傳輸已排程保留原則嗎？** <br/>
A7。 否，增量複本會根據備份排程頁面中所述的時間進行傳送。 可以保留的復原點取決於保留原則。

**Q8。 如果是長時間保留備份，則需要花很長的時間來復原的資料 （假設最舊的點）？** <br/>
A8。 否，復原最舊或最新復原點所花費的時間皆相同。 每個復原點的功能就像一個完整的復原點。

**Q9。 如果每個復原點就像一個完整的點，則其是否會影響計費備份儲存體總數？**<br/>
A9.  典型的長期保留復原點產品會將備份資料儲存為完整的復原點。 不過，這些是效率不佳的儲存體，但可以更輕鬆且更快速地進行還原。 增量複本是有效率的儲存體，但會要求您還原一連串的資料，而這會影響復原時間。 Azure 備份的唯一儲存體架構透過最佳化儲存資料以進行快速還原，並降低儲存體成本支出，可讓您魚與熊掌兼得。 這個方法可確保您有效率地使用頻寬 (輸入和輸出)，並將儲存體和復原所花時間保持在最少狀態。

**Q10。 是否有可以建立的復原點數目限制？**<br/>
A10. 編號 我們已經消除了復原點限制。 您可以建立您所需的復原點數量。

**Q11。 備份不等於我備份的資料量中為什麼要傳輸的資料量？**<br/>
A11. 已備份的所有資料會經過壓縮，且在傳輸之前會經過加密。 您可以預期 30-40% 的壓縮效益，這取決於正在備份的資料類型。

## 復原
**Q1。 我可以備份至 Azure 的資料上執行幾次復原？**<br/>
A1. 從 Azure 備份進行復原的次數沒有任何限制。

**Q2。 必須復原期間來自 Azure 資料中心的輸出流量支付費用嗎？**<br/>
A2. 編號 您的復原作業可以使用，且不收取輸出流量。

## 安全性
**Q1。 傳送至 Azure 加密的資料？** <br/>
A1. 是。 資料會使用 AES256 在內部部署伺服器/用戶端/SCDPM 電腦上進行加密，且資料會透過安全的 HTTPS 連結進行傳送。

**Q2。 位於 Azure 也會經過加密的備份資料嗎？**<br/>
A2. 是。 傳送至 Azure 的資料仍會維持在加密 (靜止) 狀態。 Microsoft 不會解密在任何時間點所備份的資料。

**Q3。 什麼是用來加密備份資料的加密金鑰的最小長度？** <br/>
A3。 加密金鑰應該至少包含 16 個字元。

**Q4。 如果我錯置加密金鑰，則會發生什麼情況？ 我可以復原的資料 （或者） Microsoft 可以復原資料嗎？** <br/>
A4. 用來加密備份資料的金鑰僅存在於客戶組織內部。 Microsoft 不會維護 Azure 中的複本，且無法存取金鑰。 若客戶錯置金鑰，則 Microsoft 無法復原備份資料。
 

## 備份快取

**Q1。 如何變更快取位置中指定的 Azure 備份代理程式？**

+ 藉由在提高權限的命令提示字元中執行下列命令來停止 OBEngine：

  ```PS C:\> Net stop obengine```

+ 將快取空間資料夾複製到具有足夠空間的其他磁碟機。 建議您從快取空間資料夾複製檔案，而非移動它們；確認備份使用新的快取空間之後，就可以移除原始的快取空間。

+ 更新下列登錄項目，其具備新快取空間資料夾的路徑：


  	| Registry path | Registry Key | Value |
  	| ------ | ------- | ------ |
  	| `HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows Azure Backup\Config` |  ScratchLocation | <i>New cache folder location</i> |
  	| `HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows Azure Backup\Config\CloudBackupProvider` | ScratchLocation | <i>New cache folder location</i> |


+ 藉由在提高權限的命令提示字元中執行下列命令來啟動 OBEngine：

  ```PS C:\> Net start obengine```

一旦利用新快取位置成功備份，您就可以移除原始的快取資料夾。
